Guidance of Writing YAML for Entity and CDO Generation

Overview​

Vizend Studio provides a feature to generate Entity and CDO using YAML format files. This document aims to describe the methods for writing YAML documents.

Basic Input​

groupId: io.vizend
appName: small-talk

groupId and appName are essential inputs that influence the package naming during code generation.

Entity Information Input​

entities:
  - name: RoomBoard
    type: Stage
    aggregate: room

Below the "entities" section, input a list of items.

Input the name and type(available types are Stage, Domain, Data, Void).

Input the name of aggregate. This is a mandatory field and influences the determination of the Entity package during generation.

Attribute Information Input​

The method for entering attribute information is applied not only for Entity but also for ValueObject and ValueGroup.

Attribute Input Overview​

entities:
  - name: Address
    type: Stage
    aggregate: address
    attributes:
      - name: city
        type: String
        comment: 도시
        example: Seoul

Below the "attributes" section, input a list of items.

Input the name. If not provided, it will generate the attribute based on the type name.

Input the "type", which can be a primitive type or a Java type name. If not provided, it will be considered as String.

If you input a "comment", it will generate a field annotation.

If you input an "example", it will be used for generating sample() method code.

Attribute Input Method​

1. Input name only​

attributes:
  - name: question

You can only specify the 'name' attribute. In this case, it will generate an attribute named 'question' with a String type.

private String question;

2. Input name and type​

attributes:
  - name: time
    type: long

The most common method is to specify 'name' and 'type'. 'Type' can include not only primitive types but also Java class name.

private long time;

3. Usage of list attributes​

attributes:
  - name: usages
    list: io.vizend.accent.domain.xray.usage.FeatureUsage

If the attribute type is List, you can define the generic type. In this case, the code will be generated as following. If 'name' is omitted, the attribute name will be determined by the type name defined in the list.

private List<FeatureUsage> usages;

4. immutable​

attributes:
  - name: time
    type: long
    immutable: true

If you set the 'immutable' attribute to true, the generated entity attributes will include the '@FieldImmutable' annotation, creating immutable attributes that cannot be modified.

@FieldImmutable
private long time;

Entity Relation Information Input​

Entity code is generated based on the relation between entities. You can define 1:N and 1:1 relation using three attributes: oneToMany, manyToOne, and oneToOne.

Definition of 1:N Relation​

If there is a one-to-many relationship between the two defined entities, you can use the 'oneToMany' and 'manyToOne' attributes.

The following example illustrates the case where RoomBoard and RoomieBoard have a 1:N relation.

Configuration for the One side Entity

entities:
  - name: RoomBoard
    ...
    oneToMany:
      entity: RoomieBoard

Configuration for the Many side Entity

  - name: RoomieBoard
    ...
    manyToOne:
      entity: RoomBoard

Entity Code Generated from the Above Configuration.

Code for the One side Entity

public class RoomBoard extends StageEntity {

    ...
    private transient List<RoomieBoard> roomieBoards;
    
    ...

Code for the Many side Entity

public class RoomieBoard extends StageEntity {

    ...
    @FieldSourceId
    private String roomBoardId;
    
    ...

In the above code, the One side Entity can conceptually have the Many side Entity objects in a List format. However, since it doesn't actually hold the objects, the corresponding field is generated as transient. The Many side Entity can have a reference to the One side, so a field of type FieldSourceId is generated.

Definition of 1:1 Relation​

When the relation between two entities is one-to-one, you can define the relation using the 'oneToOne' attribute. In this case, you can declare ownership under the 'owning' sub-attribute in one of the entities.

Configuration for the One (Owning) side Entity

    - name: Roomie
      ...
    oneToOne:
      entity: RoomieBoard
      owning: true

Configuration for the One (Not Owning) side Entity

      - name: RoomieBoard
        ...
      oneToOne:
        entity: Roomie

Generated Entity Code.

One (Owning) side Entity Code

public class Roomie extends StageEntity {

    ...
    private transient RoomieBoard roomieBoard;
    
    ...

One (Not Owning) side Entity Code

public class RoomieBoard extends StageEntity {

    ...
    @FieldSourceId
    private String roomieId;
    
    ...

In the above code, on the owner side, the entity conceptually owns the related entity but doesn't actually own it, so it's declared as a transient field.

Additionally, on the non-owning side, the entity holds the owner's entity ID in the form of FieldSourceId.

Value Object Information Input​

You can input the necessary information for ValueObject, ValueGroup, and Enum while defining the Entity. All three can be defined as attributes of the Entity or externally to the Entity.

Definition of ValueObject and ValueGroup​

Entity internal definition

     entities:
     - name: RoomBoard
       type: Stage
       aggregate: room
       attributes:
           ...
             - valueObject:
             name: LastTalk
             attributes:
               - name: roomId
               - name: name
               - name: message

The method of internal definition in Entity is to define attributes of the Entity. You can define 'valueObject' or 'valueGroup' as a sublist item under 'attributes' and specify the name and attruibutes. In this case, if you doan't input an aggregate, it is generated under the aggregate defined in the parent Entity.

Entity external definition

     valueObjects:
     - name: LastTalk
       aggregate: room
       attributes:
         - name: roomId
         - name: name
         - name: message

You can define ValueObject/ValueGroup external of the Entity. They can be defined as sublist items under 'valueObjects' or 'valueGroups'. It's important to note that you must define an aggregate. The defined ValueObject or ValueGroup will be generated under the specified aggregate.

ValueObject Code

public class LastTalk implements ValueObject {
    private String roomId;
    private String name;
    private String message;
    ...

Definition of Enum​

You can define the Enum types that used in the Entity. Same as the ValueObject, they can be defined internally or externally to the Entity.

Definition of internal Entity

     entities:
     - name: Roomie
       type: Stage
       aggregate: room
       attributes:
         - enumType:
             name: RoomRole
             entries:
               - Owner
               - Admin
               - Roomie

You can input them as sublist items under the 'attributes' of the Entity. You can list the types in 'entries'. The aggregate follows the parent Entity.

Definition of external Entity

     enumTypes:
     - name: RoomRole
       aggregate: room
       entires:
         - Owner
         - Admin
         - Roomie

You can input them as sublist items under 'enumTypes'. In this case, you must input the aggregate.

Enum Type Code

public enum RoomRole {
    Owner, Admin, Roomie
}

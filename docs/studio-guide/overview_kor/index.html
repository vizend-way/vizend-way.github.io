Studio 사용 개요

이 문서는 Studio를 처음 접하는 사람들을 위하여 전체적인 사용 방법을 알려주고 빠르게 Drama 개발을 시작하기 위한 문서입니다.

설치​

아래 화면에서 vizend studio로 검색하고 설치합니다.

Menu에서 -> File -> Settings -> Plugins -> Marketplace ->‘vizend’로 Plugin을 검색합니다. 검색된 결과에서 Vizend Studio 의 Install 버튼을 선택하여 설치를 진행합니다. 설치가 되면 IntelliJ IDE를 재시작합니다.

Backend 프로젝트 생성​

Menu에서 -> File -> New -> Project -> 나타난 대화창의 왼쪽 메뉴에서 Vizend Studio를 선택합니다. 필요한 옵션을 선택하고 Next 버튼을 선택합니다.

프로젝트의 모듈 이름을 수동으로 설정하려면 Module Prefix 란을 입력합니다. 입력하지 않으면 모듈 이름이 자동으로 결정됩니다.

Project Location을 선택하고 드라마 네이밍 규칙을 준수하여 (예, 소문자 및 하이픈(-)만 사용, 특수문자 사용 금지) Project name을 입력한 후 Create버튼을 클릭합니다.

입력된 Project name은 프로젝트 모듈 이름을 만드는데 사용됩니다.

Module Prefix를 입력하지 않는 경우 생성된 프로젝트의 모습입니다.

Module Prefix를 ‘demo’로 입력한 경우 프로젝트의 모습입니다.

Load Gradle Project버튼을 클릭하여 Gradle build script가 로드될 수 있도록 합니다. 만약 IntelliJ의 기본 JDK가 21이 아닌 경우 Gradle 로딩이 실패합니다. 이 경우 Gradle이 사용하는 JDK 버전을 21로 변경하여야 합니다.

Menu에서 -> File -> Project Structure -> Project -> SDK SDK 옵션에서 21버전을 찾아 하여 변경합니다. 만약 21버전을 찾을 수 없다면 Download JDK를 선택하여 다운로드를 진행합니다.

Version 및 Vendor 옵션을 21버전으로 선택한 후 Download버튼을 클릭합니다.

21버전 Download가 완료되면 21버전을 선택하여 OK 버튼을 클릭합니다.

Menu 에서 -> File -> Settings -> Build -> Gradle 마찬가지로 Gradle JVM에도 21버전으로 선택하여 OK 버튼을 클릭합니다.

SDK버전이 모두 21로 변경이 된다면 “Load Gradle Project” 버튼 다시 클릭하여 재실행합니다.

Entity와 Cdo 생성​

프로젝트 트리에서 Domain 모듈을 선택하고 오른쪽 마우스를 클릭한 후 ‘Create Aggregate entity and Cdo…’를 클릭합니다.

나타난 대화창에서 Entity type, Aggregate Name, Entity Name을 각각 입력하고 OK버튼을 클릭합니다.

그러면 Foo와 FooCdo 클래스가 각각 생성이 됩니다.

생성된 Foo 클래스

@Getter
@Setter
@NoArgsConstructor
public class Foo extends StageEntity {

    public Foo(String id, ActorKey requesterKey) {
        super(id, requesterKey);
    }

    public Foo(FooCdo fooCdo) {
        super(fooCdo.genId(), fooCdo.getRequesterKey());
        BeanUtils.copyProperties(fooCdo, this);
    }
...

생성된 FooCdo클래스

@Getter
@Setter
@Builder
@AllArgsConstructor
@NoArgsConstructor
public class FooCdo extends CreationDataObject {

    @Override
    public String genId() {
        // 
        return super.genId();
    }
...
@Getter
@Setter
@NoArgsConstructor
public class Foo extends StageEntity {
    //
    private String name;
    private String address;

    @FieldImmutable
    private int age;
...

Foo Entity의 속성을 입력합니다. 해당 속성이 수정되지 말아야 할 경우 ‘@FieldImmutable’ 어노테이션을 추가할 수 있습니다.

또한 ‘Gen the field enum’을 선택하면 Field의 이름들을 내부 enum 으로 정의할 수 있습니다.

Entity 속성 입력한 후 Entity의 Context Menu를 이용하여 modifyAttribute() 메소드 내용을 업데이트 할 수 있습니다.

modifyAttribute() 메소드가 업데이트 된 모습입니다.

Cdo는 도메인 Entity를 생성하기 위한 클래스입니다. Entity를 생성하기 위해서 필요한 속성이 정의되어야 합니다.

public class FooCdo extends CreationDataObject {
    //
    private String name;
    private String address;
    private int age;

    @Override
    public String genId() {
        //
        return this.code + "_" + this.name;
    }
    ...

Cdo의 속성을 입력한 후 genId() 메소드로 Entity의 Id 규칙을 정의합니다.

Cdo 사용법을 모르는 개발자를 위하여 sample() 메소드를 정의할 수 있습니다.

FooCdo 클래스에 입력한 속성을 기반으로 위와 같이 sample() 메소드를 builder 패턴으로 작성합니다.

Backend 코드 생성​

프로젝트 트리로 돌아가서 Foo Entity를 선택 후 오른쪽 마우스를 선택 후 ‘Create Backend Codes…’를 클릭합니다.

나타나는 대화창에서 ‘Create All Codes’를 선택하고 모든 코드를 생성합니다.

생성 결과를 보여주는 대화창에는 생성된 파일 개수 및 생성 라인 코드 현황을 볼 수 있습니다.

선택한 Entity와 관련된 Command, Query, Domain, Store 코드가 생성되었습니다.

자동으로 생성된 Store 인터페이스에서 제공되는 메소드 이외 추가로 메소드가 필요할 경우 "OptionStore 인터페이스"를 사용할 수 있습니다. 추가되는 메소드 네이밍은 Spring Data의 Derived Query Methods 방식으로 가능합니다.

Feature Code 생성​

Drama 프로젝트의 Feature code를 만드는 것도 매우 쉽습니다.

facade sub 프로젝트를 선택하고 오른쪽 마우스를 클릭한 후 ‘Create Feature Request…’를 클릭합니다.

만일 무언가를 등록하거나 수정하는 Feature를 구현하려면 Request Type을 CommandRequest로 선택합니다.(조회를 위한 Request Type은 QueryRequest입니다.)

Feature Name과 Sub-Feature Name은 여러 Feature Request들을 구분하기 위한 카테고리 역할을 합니다.

OK버튼을 클릭하고 Feature 코드를 생성을 진행합니다.

package io.vizend.foo.facade.feature.foo.bar.command;

...

@Getter
@Setter
@AllArgsConstructor
@NoArgsConstructor
@AuthorizedRole(FooRole.Director)
public class RegisterActiveBarCommand extends CommandRequest {

    public void validate() {
        /**
         *  FIXME : User Implementation
         * Example.
         * Assert.hasText(skuNo, "'skuNo' is required");
         * Assert.isTrue(increaseRate > 0, "'increaseRate' is greater than 0");
         */
    }
    ...

생성된 FeatureRequest 클래스입니다. package 이름 중 command 앞에 존재하는 foo, bar 는 각각 feature 및 sub-feature 이름입니다.

이후 개발자는 생성된 FeatureRequest에 Request 속성을 추가할 수 있습니다.

public class RegisterActiveBarCommand extends CommandRequest {
    //
    private String name;
    private String address;
    private int age;

    public void validate() {
        /**
         *  FIXME : User Implementation
         * Example.
         * Assert.hasText(skuNo, "'skuNo' is required");
         * Assert.isTrue(increaseRate > 0, "'increaseRate' is greater than 0");
         */
        Assert.hasText(name, "'name' is required.");
        Assert.hasText(address, "'address' is required");
        Assert.isTrue(age >= 18, "'age' must under 18");
    }
    ...

속성을 추가하고 validate() 메소드를 구현합니다.

프로젝트 트리에서 CommandRequest 클래스를 찾아서 오른쪽 마우스를 클릭한 후 ‘Create Feature Codes…’를 클릭합니다.

나타난 대화창에서 Flow Name를 결정합니다. sub-feature 이름을 기준으로 나타나며 다른 Flow Name을 원할 경우 수정 가능합니다.

OK버튼을 클릭하고 Flow생성을 진행합니다.

FlowFacade, FlowResource 그리고 Flow 코드가 생성되었습니다.

@Service
@Transactional
@RequiredArgsConstructor
public class BarFlow {

    public String registerActiveBar(String code, String name, String description) {
        /* FIXME : User Implementation */
        return null;
    }
}

생성된 Flow 클래스의 초기 모습입니다. 개발자는 이후 Flow 코드를 여기서 작성할 수 있습니다.

Frontend 코드 생성​

지금까지 작업된 backend 코드를 기준으로 프론트 코드를 생성할 수 있습니다.

프로젝트 트리에서 drama-foo 프로젝트를 오른쪽 마우스를 클릭한 후 ‘Create All Frontend Codes…’를 클릭합니다.

생성 코드가 위치할 프론트 프로젝트가 아직 생성되지 않았거나 따로 지정되지 않는 경우 이 대화창이 나타납니다. Default 프론트 프로젝트를 원할 경우 ‘Use Default Location’을 체크하고 OK 버튼을 클릭합니다.

프론트 프로젝트를 생성할 것인지 물어봅니다. Yes버튼 클릭하면 생성하게 됩니다.

프론트 프로젝트가 생성되었습니다.

나타난 화면 왼쪽 부분에서 생성할 기준이되는 Backend 파일을 선택하고 오른쪽 부분에서 생성 Target을 선택합니다.

생성 완료된 대화창을 나타나면 OK버튼을 클릭하고 생성된 코드 내용을 확인 가능합니다.

Default 위치에서 생성된 drama-foo-front 프로젝트를 따로 열어서 생성된 코드를 확인 가능합니다.

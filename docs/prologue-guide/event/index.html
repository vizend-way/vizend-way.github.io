Event Support

Vizend에서 일반적으로 사용하는 NATS JetStream 또는 Kafka 이벤트 브로커를 사용해 효과적으로 사용할 수 있는 구조를 제공합니다. 메세지 내에는 서비스에서 전송할 데이터가 담겨 비동으로 서버에 전송되고, 사용하는 서비스에서는 서버에서 이벤트 채널을 구독해 가져가 처리하는 방식으로 동작합니다.

Prologue에서는 미리 준비된 서비스 사이의 이벤트 협업 구조를 지원하고 MSA 아키텍쳐 내에 다양한 경우를 고려해 설계되어 있습니다. 아래에서는 각 이벤트 브로커 유형마다 설정 및 사용 방법을 설명합니다.

로컬 설정​

vizend:
  prologue:
    janitor:
      mode: local

로컬 개발 시 설정하는 방식으로 mode 값으로 local을 줍니다. 이벤트 브로커가 없는 상태에서 Spring Application Event 방식으로 메시지 발행 및 구독을 확인할 수 있습니다. 다른 서비스와 연계해 테스트할 경우에는 사용할 수 없습니다.

NATS 설정​

vizend:
  prologue:
    janitor:
      mode: nats 
      servers: 200.x.x.1:4222, 200.x.x.2:4222
      id: ${spring.application.name}
      subscriptions:
        - io.vizend.issue
        - io.vizend.vault

NATS JetStream 이벤트 브로커와 연결하려면 mode 값으로 nats를 설정합니다. 서버 연결은 배열로 줄 수 있습니다. id 값은 이벤트를 발행할 때 Vizend의 채널 명으로 사용됩니다. 그리고 다른 채널에서 구독할 때 Consumer Group ID로도 사용됩니다.

💡 Consumer Group ID는 이벤트를 구독할 때 처리한 메세지의 Offset을 유지하고 이후 발행된 신규 메세지를 처리할 때 꼭 필요한 값입니다. Consumer Group ID가 바뀌게 되면 Offset이 유지되지 않아 메세지를 새로 처음부터 읽거나 특정 Offset을 지정해야 할 수 있습니다.

subscriptions 는 구독할 이벤트 채널을 명시합니다. 보통 Vizend의 이벤트 채널 이름 규칙에 따라 io.vizend.issue 라고 줄 경우 io.vizend.issue-domain을 구독하게 됩니다.

Kafka 설정​

vizend:
  prologue:
    janitor:
      mode: kafka 
      servers: 200.x.x.1:9092, 200.x.x.2:9092
      id: ${spring.application.name}
      subscriptions:
        - io.vizend.issue
        - io.vizend.vault

Kafka 설정도 추상화하여 NATS JetStream 설정과 동일합니다. mode 값과 servers 값만 Kafka에 맞게 변경해주면 됩니다.

Avro 설정​

NATS JetStream과 Kafka 에서 보통 메세지는 JSON으로 구성됩니다. 메세지의 유형을 좀 더 최적화하여 메세지 사이즈를 줄이고 싶다면 Avro 메세지를 사용할 수 있습니다. 이 경우 JSON과 다르게 키와 데이터 구성이 별도의 AVRO Schema Registry 서버를 통해 유지하고 이벤트의 Payload에는 실 데이터만 바이너리로 구성해 전송하게 됩니다. 이벤트마다 동일한 구조를 가지고 있고 대량의 메세지를 처리한다면 성능에서 이점을 가져갈 수 있습니다. Vizend 드라마에서는 Prologue의 Janitor 모듈 지원을 통해 별다른 설정 없이 동일한 방식으로 사용할 수 있습니다.

Avro 메세지는 이벤트를 구성할 때 Avro Schema Subject를 지정하기 위해서는 AvroMessage 인터페이스를 상속하고 getSchema() 메소드를 구현하면 됩니다. 아래는 Avro 방식의 이벤트 클래스의 예시입니다.

@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
public class ShipboardAvroEvent extends DomainEvent implements AvroMessage {
    //
    private String id;
    private String name;

    @JsonIgnore
    private String schema;

    @JsonIgnore
    @Override
    public String getSchema() {
        //
        return this.schema == null
                ? AvroUtil.schema(this.getClass()).toString()
                : schema;
    }

    // ...
}

💡 Avro Schema JSON을 지정하지 않을 경우 간단히 Prologue의 AvroUtil.schema(..) 메소드를 이용할 수 있습니다.

설정에서는 Schema Registry 서버를 구축하고 정보를 추가해 주면 됩니다. 다음의 설정이 더 추가됩니다.

vizend:
  prologue:
    janitor:
      schema-registry-urls: http://schema_svr1:5000, http://schema_svr2:5000

이 설정 외에 이벤트 발행 및 구독에서는 일반 JSON과 Avro 메세지를 사용할 때 다른 차이점은 없습니다.

Vizend 이벤트​

Vizend 이벤트는 3가지 유형이 있습니다. DataEvent는 도메인 모델의 엔티티 단위의 등록, 수정, 삭제 내용을 기록하고 이벤트 소싱 방식의 아키텍처에서 주로 사용되고 비지니스 로직에서는 사용하지 않습니다. 사용자 요청을 분석하기 위한 RequestEvent는 X-ray 서비스 분석에서 사용되고 로직 레벨에서 사용하지 않습니다. 보통 서비스 사이의 이벤트 처리는 DomainEvent 를 사용합니다.

이벤트채널 이름설명
RequestEventid + -requestREST API를 통한 사용자 요청, 응답을 기록하며 X-ray에서 분석을 위해 사용됩니다. Command와 Query 사용 시 내부에서 자동 처리됩니다.
DataEventid + -data엔티티 단위의 라이프 사이클을 모두 이벤트로 기록합니다. 이벤트 Replay를 통해 모든 엔티티 상태를 재구성할 수 있습니다. domain 모듈 내의 logic 에서 사용되고 사용자가 만들지 않습니다.
DomainEventid + -domain서비스에서 서비스 사이에 이벤트 처리를 하기위한 목적입니다. 비지니스 로직에서 사용자가 명시적으로 구성하고 발행 및 구독을 통해 협업할 수 있습니다.
NamedChannelEvent사용자 지정사용자가 지정한 채널 이름을 주어 특수한 목적으로 사용합니다. Vizend에서는 개발 테넌트에게 인스턴트 메세지를 전송할 때 사용합니다. getChannelName() 메소드를 구현해 이름을 지정할 수 있습니다.

이렇게 3가지 이벤트는 사용하지 않을 경우 다음의 설정을 통해 발행 여부를 선택할 수 있습니다.

vizend:
  prologue:
    janitor:
      event:
        data:
          enabled: true
          partition: 2
        domain:
          enabled: true
          partition: 2
        request:
          enabled: true
          partition: 2
        namedChannel:
          enabled: true
          partition: 2

위와 같은 기본값을 가지고 있고 사용하지 않을 경우 각각의 enabled 값을 false로 주면 채널을 만들지 않고 이벤트 브로커로도 발행되지 않습니다. 사이트의 프로젝트 필요에 따라 선택할 수 있습니다.

이벤트 발행 및 수신 방법​

DataEvent, RequestEvent는 개별 서비스 경계를 넘어갈 수 없습니다. 자신의 서비스(In-service) 내에서 Spring Application Event 방식으로 수신해 추가 처리를 하는 용도로만 확장할 수 있습니다. 일반적으로 DomainEvent만을 사용하고 다음과 같이 이벤트를 발행하고 수신할 수 있습니다.

이벤트 발행​

DomainEvent를 확장해 사용자 이벤트를 생성합니다. 이벤트 객체는 event 모듈에 작성되고 별도 Library로 Nexus 등의 저장소에 Publish 됩니다. 다른 서비스에서 이 Library를 종속성으로 걸어 이벤트 수신에서 사용합니다.

@Getter
@Setter
@NoArgsConstructor
@AllArgsConstructor
public class FooEvent extends DomainEvent {
    //
    private String id;
    private String name;
    private long time;

    @Override
    public String toString() {
        //
        return toJson();
    }

    public static FooEvent fromJson(String json) {
        //
        return JsonUtil.fromJson(json, FooEvent.class);
    }

    public static FooEvent sample() {
        //
        return new FooEvent("foo001", "Foo Group", System.currentTimeMillis());
    }

    public static void main(String[] args) {
        //
        System.out.println(sample().toPrettyJson());
    }
}

💡 이렇게 만들어진 DomainEvent 객체는 library 모듈로 묶여 넥서스에 배포되고 이 이벤트를 구독하는 쪽에서 Dependency를 걸어 사용합니다. Studio를 통해 프로젝트를 구성하면 ooo-event 모듈이 만들어지는데 이를 위한 서브 모듈입니다.

다음은 Feature 사용자 로직에서 이벤트를 발행하는 코드입니다. EventProxy 라는 객체는 Prologue를 통해 기본 생성되는 Bean 입니다.

@Component
@Transactional
@RequiredArgsConstructor
public class FooFlow {
    //
    private final EventProxy eventProxy;

    public void publish() {
        //
        eventProxy.publish(FooEvent.sample());
    }
}

이벤트 구독​

이벤트를 구독하는 서비스에서는 nats 또는 kafka 별로 설정을 하고 구독을 원하는 서비스를 subscriptions에 추가해 구독한 상태여야 합니다. 이벤트 브로커 쪽로부터 처리할 메세지가 발행되어 있으면 Prologue에서 해당 메세지를 수신해 동일한 DomainEvent 객체로 변환해 구독하는 쪽의 지정된 메소드가 호출됩니다. @EventHandler로 명시된 메소드 내에 수신 후 처리할 로직을 구성하면 됩니다.

@Component
public class FooEventHandlers {
    //
    @EventHandler
    public void on(FooEvent event) {
        //
        // 처리할 로직 코드 작성...
    }
}

💡 이벤트를 수신해 처리하는 클래스는 facade 모듈에 위치하고, 기능 또는 서비스 단위로 핸들러를 구성해 처리하는 방식입니다. 핸들러 내에서 직접 모든 로직을 구현하기보다는 feature 쪽의 Seek 또는 Flow 로직을 호출해 사용하는 방식이 좋습니다.

사용자 이벤트 채널 사용​

이벤트 발행 및 구독은 Vizend 아키텍처 내에서는 서비스 패키지에 따라 서비스패키지-domain 또는 서비스패키지-data 규칙의 채널명이 사용됩니다. 하지만 BC(Bounded Context) 외부 서비스와 이벤트를 연계할 경우 내부 규칙을 사용하기에 적합하지 않습니다. DomainEvent 또한 공유가 어렵습니다. 이 경우 다음과 같이 사용자 이벤트 채널 및 이벤트 객체 변환을 사용할 수 있습니다.

사용자 이벤트 발행​

BC 내부에서의 이벤트 발행에서 사용하는 EventProxy 에서는 publish(..) 메소드를 Overloading하여 채널 이름을 지정해 발행할 수 있습니다. 다음은 그 예제입니다.

@Component
@Transactional
@RequiredArgsConstructor
public class FooFlow {
    //
    private final EventProxy eventProxy;

    public void publish() {
        //
        eventProxy.publish("my-channel", FooEvent.sample());
    }
}

위 예제에서는 동일한 이벤트를 my-channel 이라는 이름으로 발행합니다. 물론 이 때에도 발행 메세지는 DomainEvent를 구현한 객체를 사용하면 됩니다.

사용자 이벤트 구독 및 변환​

BC 외부에서 공유한 이벤트를 구독해 처리하는 것도 가능합니다. 외부 이벤트는 자신의 채널 이름을 가지고 있는데 이를 수신하는 방법은 subscriptions 구독 목록에 ^로 시작하는 채널 이름을 사용하면 됩니다.

vizend:
  prologue:
    janitor:
      mode: kafka 
      servers: 200.x.x.1:9092, 200.x.x.2:9092
      id: ${spring.application.name}
      subscriptions:
        - ^my-channel
        - ^user-channel

구독한 메세지는 Vizend의 Prologue 모듈에서 JanitorStreamEvent로 1차 변환되기 때문에 이를 이용하면 외부 이벤트 객체도 간단히 사용할 수 있습니다.

@Component
public class FooEventHandlers {
    //
    @EventHandler
    public void on(JanitorStreamEvent event) {
        //
        // event.channelName, event.payload를 확인해 처리 로직 사용자화 구성
    }
}

참고로 JanitorStreamEvent는 다음과 같은 구조를 가지고 있습니다.

┌──────────────────────────────┐ 
│     JanitorStreamEvent       │ 
├──────────────────────────────┤ 
│ Map<String, String> headers; │ Event header (Tracing Meta Data)
│ String id;                   │ Event identifier (TinyUUID)
│ OffsetDateTime time;         │ When the event was issued
│ JanitorEventType type;       │ Event types (Data, Domain, Request)
│ String payloadType;          │ Payload type string
│ String payload;              │ Payload data (JSON)
│ String routeKey;             │ Routing key for partitioning
│ String subject;              │ Event subject
│ URI source;                  │ Event source URI (*Reserved)
│ String channelName;          │ Event Channel Name
└──────────────────────────────┘

다중 이벤트 채널 수신​

여러 이벤트 채널을 모두 수신해야 할 경우 Wildcard 방식의 subscriptions 표기를 지원합니다. 가령 io.vizend.metro, io.vizend.valut, io.vizend.gallery 이벤트를 모두 수신해야 할 경우 다음과 같이 설정할 수 있습니다.

vizend:
  prologue:
    janitor:
      mode: kafka 
      servers: 200.x.x.1:9092, 200.x.x.2:9092
      id: ${spring.application.name}
      subscriptions: ^io.vizend.*
